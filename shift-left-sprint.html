01 Core Practices
📐 Planning
Smaller, Crystal-Clear Stories
Break epics into laser-focused stories with a single, testable outcome. Each story should be completable within 1–2 days, leaving no ambiguity in scope or acceptance criteria.
🤝 Refinement
Devs Refine with PO & SM
Developers actively co-refine sub-tasks alongside the Product Owner and Scrum Master. This ensures technical feasibility, shared understanding, and no hidden surprises mid-sprint.
3-amigos before every story
🎯 Sprint Kick-off
Start with a Clear Sprint Goal
Every sprint opens with a written, agreed-upon goal that the whole team can articulate. This north-star guides prioritisation decisions when the unexpected hits mid-sprint.
Written + visible at all times
⚡ Development
Faster Implementation
Smaller stories mean shallower context-switching and faster flow. Developers can enter deep focus, implement fully, and hand off the same day — eliminating multi-day WIP accumulation.
🔀 Code Review
Small & Mid PRs = Faster Reviews
PRs scoped to a single concern are reviewed thoroughly in minutes, not hours. Reviewers stay engaged, catch more issues, and approve confidently — no rubber-stamping of 800-line diffs.
Target: <500 lines per PR
🧪 Testing
Early Testing via Preview Envs
Once a PR is reviewed and the code is stable, it deploys to a preview environment. QA, PO, and designers validate on real, reviewed code — catching issues early before they reach staging.
🔁 Staging
Faster, Multiple STG Deployments
After preview sign-off, code is promoted to staging where full regression testing runs. Frequent STG deployments keep the environment fresh and give QA a stable, PROD-like surface to test against.
🚀 Production
Manual PROD Deployment
After regression passes on STG, the team triggers a deliberate manual PROD deployment. The team owns the release decision — intentional, controlled, and confidence-backed by prior testing.
02 The Shift Left Sequence
Phase 1 · Before Sprint
Quality-first Refinement
Acceptance criteria written, edge cases discussed, tasks broken down. DoR (Definition of Ready) gates stories from entering the sprint. Automated dependency checks flag blockers before planning day.
Phase 2 · Sprint Day 1
Aligned Kick-off & Goal Setting
Team aligns on the sprint goal, stories are assigned with clear ownership. Each dev walks out of planning knowing exactly what "done" looks like for their slice.
Phase 3 · During Sprint
Continuous Flow: Code → Review → Preview → Test
Developers submit PRs for review. Once approved and code is stable, it deploys to a preview environment where QA, PO, and designers validate. Testing happens on reviewed code — not raw branches — keeping the feedback loop tight and the signal trustworthy.
Phase 4 · Staging
Regression Testing on STG
After preview sign-off, code is promoted to staging where full regression testing runs. STG mirrors PROD closely, giving QA confidence that nothing is broken before release.
Phase 5 · End of Sprint
Manual PROD Deployment
Once regression passes on STG, the team triggers a deliberate, manual PROD deployment. Intentional and controlled — the team owns the release decision. Sprint review showcases real shipped value.
03 Our Rollout Plan
Step 1 · Now — Active
Start Creating Smaller Stories
The team commits to breaking stories down before they enter the sprint. Each story should have a single, clear outcome that one dev can own end-to-end. This is the highest-leverage change — everything else flows better when the unit of work is small and well-defined.

✦  What good looks like: stories that can be built, reviewed, and tested within 1–2 days without ambiguity.
Step 2 · Next Sprint
All Devs, PO & SM Refine Together
Once story sizing improves, the full team joins refinement — every dev, the PO, and the SM in the same session. Devs bring technical depth, PO brings priority and context, SM facilitates and removes blockers. No story enters the sprint that the team hasn't collectively understood.

✦  What good looks like: zero "surprise" blockers mid-sprint. Every dev could pick up any story.
Step 3 · Following Sprints
Layer in the Rest of the Framework
With solid story sizing and shared refinement in place, the team can progressively adopt preview environments, tighten PR discipline, and establish the Code → Review → Preview → STG Regression → PROD flow. Each step builds on the last — don't rush ahead.
04 Story Breakdown — Real Example

Below is how one large story — "Approve/Deny Pending Applications via Provenir API" — gets split into 2 independently deliverable stories.

S-01
API · Infra
Provenir Status Update API — Endpoint, Validation & Error Handling
As a: Fidem administrator
I want to: have a secure, documented API endpoint that Provenir can call to submit application status updates
So that: status changes are authenticated, validated, and rejected gracefully before any downstream processing begins
Description
Build the POST endpoint Provenir uses to update application status. Covers API key/secret authentication, Swagger documentation, pre-flight validation checks (email, existing card, SSN duplicate), structured error responses back to Provenir, and blocking any change attempt on already-Approved applications. Failure emails are sent to aress-servicing for every error.
Notes / Considerations
  • SSN duplicate check is behind a feature flag — must handle both enabled and disabled states
  • Failure emails must include API timestamp, LOS ID, and a direct link to the application in CMT
  • Swagger docs and API key/secret management process must be completed before S-02 can begin
  • Already-Approved applications must be blocked regardless of the requested target status
Acceptance Criteria
  • Endpoint authenticates via API key/secret and returns 401 for invalid credentials
  • EMAIL_VERIFICATION_FAILED returned when no DbUser found for {productId}-{email}
  • ALREADY_EXISTS returned when user already has a non-null fiservAccountId
  • SSN_DUPLICATE returned when SSN check is enabled and a duplicate is found; flow continues normally when flag is off
  • Any status change request against an Approved application returns a clear error and leaves CMT unchanged
  • Failure email sent to aress-servicing with timestamp, LOS ID, and CMT link for every error case
  • Swagger documentation published and accurate
S-02
Core Flow
Provenir Status Update API — Approved & Denied Downstream Flows
As a: Fidem administrator
I want to: have the system automatically complete all downstream account creation and notification steps when an application is approved or denied via Provenir
So that: applicants are onboarded correctly without manual intervention and all systems stay in sync
Description
Implements all downstream processing triggered by a valid status change. For Approved transitions (from Pending or Denied): books the account to Fiserv, stores returned IDs in CMT, creates and transmits PAN 16 to SYW, handles offer codes, and sends the welcome email. For Pending → Denied: updates CMT status only. All failures are logged and escalated. CMT displays the full API call history per application.

Notes / Considerations
  • Denied → Approved must run the exact same flow as Pending → Approved — the starting state is the only difference
  • Offer code handling only applies if the user applied with an offer code; no action if they didn't
  • No email or Fiserv calls for the Denied flow — denial notice is sent via physical mail
  • CMT must always reflect the most accurate completed state even if a downstream step fails mid-flow
  • Depends on S-01 being complete and the API endpoint being live
Acceptance Criteria
  • Pending → Approved: CMT status updated to Approved, account booked to Fiserv, Fiserv Account ID / External Customer ID / Affinity ID stored in CMT
  • Pending → Approved: PAN 16 created at Fiserv and securely transmitted to SYW
  • Pending → Approved: applied offer code marked as "Applied", all other offer codes for that user deactivated (only when offer code was used)
  • Pending → Approved: welcome email with required disclosures sent to the applicant
  • Denied → Approved: same outcome as Pending → Approved across all steps
  • Pending → Denied: CMT updated to Denied, no email sent, no Fiserv or SYW calls made
  • Any downstream failure is logged with an error status and a failure email sent to aress-servicing with timestamp, LOS ID, and CMT link
  • CMT API results view shows all calls for an application in order: original submission + every status change (3 entries for Pending→Denied→Approved, 2 for Pending→Approved and Pending→Denied)
05 Supporting Principles
01
Definition of Ready (DoR)
Stories only enter the sprint backlog when they meet the DoR: clear AC, designs attached, dependencies resolved, and estimated. No more vague tickets causing mid-sprint blocking.
02
Trunk-Based Development
Short-lived branches merged frequently to main. Feature flags decouple deploy from release. Reduces long-lived branch merge hell and enables continuous integration in practice.
03
Automate the Quality Gates
Linting, formatting, unit tests, coverage thresholds, and security scans run on every commit. Humans review logic and design, not style. CI/CD is the quality safety net, not a release blocker.
04
Observability from Day One
New features ship with logging, metrics, and alerting wired up. No flying blind post-release. If something breaks in PROD, the team knows within minutes — not after a customer reports it.
05
Retrospect on Flow Metrics
Track lead time, cycle time, PR review time, and deployment frequency. Surface bottlenecks with data. Retros become improvement sessions, not complaint sessions.
