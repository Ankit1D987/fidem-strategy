4-Stage Pipeline

Development & Testing Flow

🌿

Feature Branch Created

Developer picks up a ticket from the sprint backlog and creates a dedicated feature branch. Work begins in isolation from the main codebase.

💻

Code Written & Committed

Developer completes the ticket and commits changes to the feature branch.

🔍

Pull Request Opened → Code Review

PR is raised. Another dev reviews the code and adds comments. Developer addresses all review comments. PR is approved ✅

🟣

Testing on Preview

Once PR is approved, QA performs full AC-based feature testing.

Full AC Testing · QA + PO
🔀

Branch Merged → PR Closed

After QA sign-off on Preview, branch is merged into DEV. Preview environment is destroyed. ⚠ To be confirmed with Kevin.

🟡

Smoke Test on DEV

All sprint features are together on DEV. QA runs a lightweight smoke test to catch integration issues — not a full re-test of each feature.

Smoke Test · Integration Check · QA
🟢

Full Regression on STG

Stable DEV build promoted to STG. QA runs full regression suite. Bug fixes only — no new features. Sign-off required before PROD.

Full Regression · Release Readiness · Nick
🚀

Deploy to PROD

After STG sign-off, release is deployed. Team monitors logs, error tracking, and user feedback.

Environment Responsibilities

Preview
Feature Isolation
  • Triggered by PR approval
  • Per-feature, isolated URL
  • Full AC verification
  • QA + PO sign-off
  • Destroyed after merge
DEV
Integration
  • All sprint features merged
  • Smoke test only
  • Check integration health
  • May be unstable
STG
Regression
  • Mirrors PROD closely
  • Full regression suite
  • No new features here
  • Bug fixes only
  • Nick sign-off
PROD
Live
  • Real users
  • Monitor only
  • Error tracking active
  • No direct hotfixes
  • Go through STG first

Testing Summary

Environment Testing Type Done By Goal
Preview Full AC-based feature testing QA + PO Verify the ticket works as specified
DEV Smoke test only QA Confirm integration didn't break anything
STG Full regression Nick Release readiness sign-off
PROD Monitoring only Team Catch live issues early

Why We Cannot Skip DEV Testing

⚡
Integration Conflicts
Feature A and B may each work on Preview individually but conflict once merged — shared state, overlapping API calls, or route clashes.
🧩
Shared Component Breakage
One ticket may modify a reusable component (button, modal, form) that other tickets depend on — invisible on Preview, visible on DEV.
🔌
BE / FE Subtask Misalignment
BE and FE are merged separately. API contracts can drift — field names or response shapes that only break on full integration.
💾
Data Dependency Issues
One feature's data changes can affect another feature's flow — a ticket updating user state may break another ticket's logic.
🛡️
Keeps STG Clean
Skipping DEV smoke testing means integration bugs surface during STG regression — wasting QA effort and delaying the release.
⚠️
Bottom Line Preview tests features in isolation. DEV tests them together. Skipping DEV means skipping the only place where integration bugs can be caught before they pollute STG and delay your release.

Golden Rules

Rule 01
Test After PR Approval, Not Before
QA only tests on Preview after the PR is fully approved. Testing on a PR with open comments wastes effort — the code may still change.
Rule 02
Smoke Test DEV — Never Skip It
DEV gets a smoke test, not a full regression. Features were tested in isolation on Preview — DEV is the only place to catch integration bugs before STG.
Rule 03
STG Must Mirror PROD
STG should match PROD as closely as possible — same data shape, same configs. If STG passes, PROD should too.
